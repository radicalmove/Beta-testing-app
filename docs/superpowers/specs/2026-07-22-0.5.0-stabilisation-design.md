# Beta Testing App 0.5.0 Stabilisation Design

**Date:** 22 July 2026  
**Status:** Approved for implementation  
**Target:** Extension and server version 0.5.0

## Goal

Turn the 0.4.69 pilot candidate into a stable 0.5.0 retest build by fixing every known pilot issue as one coordinated release. Preserve the behaviours already confirmed during manual testing, including the single toolbar, Moodle and Rise comment creation, grouped course comments, working Moodle navigation, and course-wide Previous/Next order.

## Scope and approach

The release is organised around shared state and permissions rather than a sequence of isolated visual patches:

1. Server-side roles, visibility, sharing, attachments, and idempotent state changes.
2. A single client-side projection for comment ordering, filtering, navigation, and refresh.
3. Consistent toolbar, list, marker, thread, modal, and help presentation.
4. Course-scoped reviewer sessions and course-first administration.
5. Automated regression coverage plus a focused manual retest checklist.

No email notifications, public extension-store distribution, or Moodle plugin is included in this release.

## Authoritative data and permissions

The server remains authoritative for identity, role, course membership, thread visibility, and write permissions. The extension must render the role returned by the server rather than infer a role from the thread type or the viewer.

- Administrators, Learning Designers, and Digital Course Developers can see all course feedback, reply, resolve, delete where permitted, and share a beta-tester thread with selected SMEs.
- Beta testers see their own threads and course-team replies to those threads. They do not see SME-only replies.
- SMEs see their own threads, normal SME-visible feedback, and beta-tester threads explicitly shared with them. They can reply to shared beta threads; those replies are visible only to SMEs assigned to the thread and to Administrators/LDs/DCDs.
- A thread created by an SME is already SME-visible and therefore must not show a redundant “Share with SME” control.
- Repeating an already-applied status action is idempotent. It must refresh to the authoritative status instead of returning a visible 422 error.

The operation matrix is explicit:

| Operation | Beta tester | SME | LD/DCD | Administrator |
|---|---|---|---|---|
| Create a thread | Yes | Yes | Yes | Yes |
| Edit originating comment | Own only | Own only | Own only | Own only |
| Reply to beta thread | Own thread only | Only when explicitly assigned | Any | Any |
| Reply to SME/non-beta thread | No, unless own | Any visible SME thread | Any | Any |
| Resolve/reopen/change status | No | No | Any visible thread | Any visible thread |
| Delete thread | Own only | Own only | Any visible thread | Any visible thread |
| Share beta thread with SMEs | No | No | Yes | Yes |
| Attach to originating comment | Own comment | Own comment | Own comment | Own comment |
| Attach to a reply | When allowed to create that reply | When allowed to create that reply | When allowed to create that reply | When allowed to create that reply |

Deleting a thread cascades to its replies and attachments. Attachment removal follows the existing uploader/lead permissions and does not grant a user permission to edit another participant's text.

The share control is labelled **Share with SME**. It appears beneath the Next button, matches its width, and uses the approved burnt-orange selected/unselected treatment. Expanding it shows one row per eligible SME, the standard square tick control, and the shared save icon button. An eligible SME has an approved SME course membership in the same course. Saving replaces the complete selected-SME set; it is not additive. An unassigned SME immediately loses access to the thread and its private conversation. Their historical replies remain stored for audit and remain visible to Administrators/LDs/DCDs and SMEs who are currently assigned, but not to the beta tester or the removed SME. The server returns the complete updated sharing state.

## Comment projection and ordering

The extension constructs one canonical, viewer-specific projection of comments. That projection drives:

- the right-hand comments list;
- “Comment x of y”;
- Previous and Next;
- course/page and open/resolved filters;
- marker and highlight visibility;
- Jump to page choices.

Within a loaded page, physical anchor order takes precedence over creation time. Across the course, Moodle section/activity order and saved embedded activity order take precedence. Resolved comments are excluded from the open projection and open Previous/Next sequence; resolved markers, highlights, and open thread popovers are hidden while Open is selected. The inverse applies while Resolved is selected.

When an anchor cannot be recovered, the list keeps the comment available but shows a clear in-app recovery message instead of silently doing nothing. Administrators/LDs/DCDs retain delete access for genuinely stale feedback.

## Navigation

### Moodle

Selecting a list item or using Previous/Next navigates to the correct Moodle page, recovers the anchor, scrolls it into a safe visible area, marks its comment icon active, and opens its thread. The course panel remains open during navigation.

### Rise/SCORM

Deep navigation is best-effort because Rise may render asynchronously without changing the outer Moodle URL.

1. Navigate to the saved Moodle SCORM launch URL, preserving required launch parameters such as `scoid`.
2. Wait for the supported embedded frame to announce readiness.
3. Retry lesson/interaction and anchor recovery while Rise renders or changes lesson state.
4. Scroll to and open the marker automatically when recovered.
5. If recovery reaches its bounded timeout, leave the user on the correct SCORM activity and show an in-app message explaining that the precise point could not be restored and the comment remains available in the list.

Jump to remains available in both Whole course and Current page views because it is a navigation control, not a list filter. It must use valid Moodle launch URLs and never open raw SCORM package content outside the LMS.

## Live synchronisation and state preservation

Visible course data refreshes periodically and when the tab regains focus. Refresh must merge server changes without replacing active drafts, edit/reply composers, panel state, scroll position, or the selected thread. Fresh replies, status changes, and new visible comments appear without a full page refresh.

The open/collapsed panel state remains course-specific. Entering or cancelling marker mode does not collapse the comment panel or leave stale placement instructions.

## Marker, highlight, and thread presentation

- An active comment marker uses burnt orange; inactive markers use teal. Marker-placement candidates do not borrow the destructive red treatment.
- Saved text highlights use a subtle pale yellow with dark inherited text and sufficient contrast across multi-line selections.
- Thread popovers avoid the comments panel by moving left when necessary and remain attached to their anchor while scrolling. They hide when the anchor is outside the usable viewport rather than sticking to the header or screen edge.
- Comment previews use available width up to the resolve/delete controls, wrap to at most two lines, then truncate.
- Thread messages are readable conversation cards. The initiating comment is visually distinct from replies; each participant receives a stable, accessible colour treatment and replies alternate indentation by speaker direction. Names and true server roles remain visible. Four or more participants remain distinguishable without relying on colour alone.
- Transient refresh never collapses a reply into a narrow column or reflows the navigation buttons.

## Status, delete, and confirmation behaviour

Resolve from a popover uses the same in-app confirmation modal as resolve from the list. After confirmation, the square control shows the approved green pen-style tick for approximately three seconds before the thread disappears from Open. Delete uses an app modal—never the browser’s native confirm—and clearly states that the thread, replies, and attachments will be removed.

## Attachments

JPEG, PNG, PDF, and Word attachments up to 10 MB work for creation, editing, and replies. Upload commands carry the same authoritative course/thread/comment context used by the API. Cancelling or successfully retrying clears any transient upload error. Errors are written for users, not internal command names such as `UPLOAD_SCREENSHOT`.

## Sessions and administration

The toolbar identifies the signed-in user and role and provides a clear course-scoped sign-out/switch-user action. A normal browser may remember the course reviewer; an incognito window must not silently inherit a reviewer from another browser context except where Chrome itself explicitly preserves extension storage. Signing out returns to the course sign-in chooser without affecting other course identities.

Administration presents course selection before reviewer/invitation management. The selected course name is prominent and the user/invitation lists update for that course, preventing codes from being copied from the wrong course.

## Help and language

The help modal is wider and uses concise sections covering text comments, markers, Moodle versus Rise behaviour, filters, navigation, conversation/reply visibility, attachments, resolve/delete, sharing with SMEs, sign-out/switch user, and recovery messages. It must not require unnecessary scrolling at common laptop sizes.

## Verification and release

Every defect receives a failing automated test before its implementation where the behaviour can be exercised deterministically. Server tests cover the permission matrix, idempotence, roles, sharing, sessions, attachments, and admin routes. Extension unit/E2E tests cover projection ordering, filters, refresh, modals, layouts, marker/highlight styling, and Moodle/Rise navigation. Existing 170 server tests and the full extension suite remain green.

The final build is versioned 0.5.0, packaged using the existing pilot release scripts, checked for manifest/runtime consistency, and accompanied by a manual retest checklist mapped to P01–P25.
